Executive Summary
Brute force attack is a common technique used for password cracking, where attackers employ a refined strategy to bypass traditional security measures. This report demonstrates an attack on the (D)DVWA login page at the medium level, utilizing the Burp Suite Intruder to exploit the login page. The DVWA is an insecure and frequently targeted simulation in a penetration test. The solution incorporates a custom login rate-limiting mechanism within the DVWA medium.php, where a session-based lockout system logic is used. Fail2Ban is configured to monitor Apache logs and automatically ban IP addresses for suspicious logins.  
All these changes show a practical strategy that is aligned with the NIST Cybersecurity Framework NIST SP 800-53 (AC-7, AU-6) and (PR.AC-7, DE.CM-7), which enforces restricted login attempts, active monitoring, and audit of unsuccessful logins CITATION NIS18 \l 1033  (NIST, 2018). It also complies with the ISO/IEC 27001:2022 control framework controls, including A.5.15 (Authentication Information)  CITATION ism221 \l 1033 (isms, 2022) and A.8.12 (Event logging) CITATION ism22 \l 1033  (isms, 2022), which ensures robust access control with real-time threat detection. 
Threat Analysis and Research
Problem Statement:
A brute-force attack uses a trial-and-error process to crack passwords, gain unauthorized access, and encrypt keysCITATION For253 \l 1033  (Fortinet, 2025).  This attack is still one of the prevalent attacks, especially in the web application environment. Research shows a 12% rise in brute-force cyberattack techniques CITATION Sec24 \l 1033  (Staff, 2024).  OWASP warns that login interfaces are likely to be targeted if they attempt unlimited attempts, as attackers these days automate billions of guesses for success CITATION Esh25 \l 1033  (Esheridan, 2025). Industries that are exposed to login interfaces like financial services, government, and healthcare are most likely to be attacked. 
Threat Impact: Real World Example
In January 2025, Shadowserver Foundation observed a rise in brute force attacks where web login pages of various network security devices, such as Palo Alto Networks, SonicWall, and Ivanti, were targetedCITATION Ern25 \l 1033  (Naprys, 2025). There were 2.8 million daily unique IP source addresses targetedCITATION Bur25 \l 1033  (Burt, 2025).  Another example, 2017 brute force on WordPress, where the XML-RPC multicall method was used to submit hundreds of login attempts in a single request. Over 1900,000 WordPress sites were affected within days, which shows a lack of IP rating limit and disabling unnecessary APIs. 
Current Security Gaps
Although the awareness organizations remain vulnerable to brute force due to inadequate implementation of the defensive controls:
Lack of Rate Limiting or Account Lockouts: An Attacker can attempt multiple credential combinations as the system is not hardened with proper request throttling and lockout policy. 
Weak or Reused Credentials: Most of the time, users are highly reliant on predictable passwords like admin123, which are very easy to crack.
Inconsistent MFA: MFA helps render brute force attempts, though the username or password is guessed correctly.
Poor Logging and Monitoring: Most of the organizations lack centralized logging or auditing, which results in undetected brute force attempts.
Captcha: This helps to verify and prevent bots from continuing brute-force attacks when two or three failed attempts are made.

Security Solution Design
3.1 Proposed Solutions:
To secure the login page against any kind of brute force attack, a multi-layered security tactic was deployed. 
Fail2Ban (IP Ban on Repeated Failures)
This is deployed on the DVWA server, which is used to automate the banning of malicious IPs after detecting multiple failed login attempts. The custom DVWA log file can be watched at (/var/log/dvwa_login.log).
PHP session-based lockout and logs: 
This logic is used to ban a specific user after multiple failed attempts. New modifications are made where the failed login attempts are tracked through PHP $_SESSION ['attempts']. Every failed attempt is logged to /var/log/dvwa_login.log. It also introduces a sleep () delay after 3 failed attempts and shows an error message like “Too many failed attempts”. Custom logging and log monitoring are implemented for brute-force detection. 
Apache2: 
This helps to access the log and helps them to track in real-time visibility.

3.3 Workflow
Brute Force Attack: An attacker uses tools like Burp Suite or Hydra from Kali Linux and conducts a brute-force attack where multiple login requests are sent with different credentials to the DVWA login form, i.e., medium.php. 
DVWA Process Request: All login requests are handled by the medium.php script. When a PHP session is started or resumed, the number of failed login attempts is tracked in $_SESSION[‘attempts’]. 
PHP session Lockout Triggered: After each failed login attempt it displays “Login failed#1”, etc., and after the 5th wrong attempt, it delays using sleep (5). The failed login is logged in the custom file /var/log/dvwa_login.log. 
Fail2ban Detection: It monitors the custom log where failregex is used to detect repeated failures from the same IP. As the attackers fail 5 attempts in 10 minutes using attack through tools like Hydra, the firewall rule is triggered to ban the IP using iptables.
Blocked Attacker at Network Layer: The attacker can't connect or open DVWA anymore.
User Recover and Feedback: The delay is short, and after the session reset the attempt counter resets. Both banned IPs via Fail2Ban are automatically unbanned after the configured ban time.

Configuration Steps:
Fail2ban Setup
Step 1: Installation Command: sudo apt install fail2ban
Step 2: Jail Config
Command: sudo nano /etc/fail2ban/jail.local
Here, the ignoreself = makes fail2ban ban own system IP (127.0.0.1) as local attempts are made from Burp Suite on localhost. Ignoreip = is empty because no IP are ignored. Once an IP exceeds the entry limit it will be banned for 10 minutes. It also looks at a 10-minute window, so if 5 failed attempts are allowed if exceeds the IP gets banned within 10 minutes. Custom jail named [dvwa-brute] is enabled so that it can monitor DVWA by checking the Apache log file /var/log/apache2/access.log for suspicious login attempts.  If there are 3 failed login IP is banned for 1 hour, and it uses filter file dvwa.brute.conf for failed login pattern detection, which is applied to both HTTP and HTTPS traffic. 
Step 3: Code Changes and Edit
Command: sudo mousepad /etc/fail2ban/jail.conf
Here the lines 85, 87, and 92, 95 have been removed to avoid config conflicts. 
Step 4: Filter Config
Command: sudo mousepad /etc/fail2ban/filter.d/dvwa-brute.conf  
Here, the failregex defines a pattern to match an HTTP GET request that targets the vulnerable DVWA brute-force URL/DVWA/vulnerabilities/brute. The <HOST> acts as a placeholder used by Fail2Ban to extract the IP address of the attacker from Apache logs. The ignoreregx is empty as no request are excluded from detection.

Limitations
Despite the implementation of proven effectiveness in the lab setup, several constraints are still identified. The list is provided below:
Session Scope: The session-based tracking can be easily bypassed with the cookie clearer or proxy rotation. 
Fail2Ban limitation: As it is reactive and log-based, the attacks cannot be detected in the encrypted payloads until the attacker vector is revealed. There is no CAPTCHA or MFA integrated.
Single Host Deployment: This test is on the DVWA server, but in the real world, a load-balanced application requires log collection and distributed Fail2ban and firewall control. 
Future Improvement
Captcha: Due to time limitations, Captcha was not implemented. But introducing it after a few failed attempts would have blocked unsophisticated bots.
Stronger Password Hash: Modern slow hashes like bcrypt/argon2 need to be implemented so that even if attackers obtain hashes, the slow hashing algorithm will buy time against offline attacks.
AI enhancement: The current system flow uses statistical thresholds, meaning after 5 attempts, a 5-minute ban. This can be improved by integrating machine learning models like One-Class SVM that can dynamically assess patterns of login requests. 
Scalability: Although this works well on a small-scale testbed Apache with ModSecurity can be used across all services. A SIEM system like Wazuh or Splunk can be enabled for real-time alerts from Fail2ban and ModSecurity across the distributed environment.
